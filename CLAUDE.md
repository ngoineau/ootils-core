# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

Target-architecture notes may appear below. Verify runtime reality before treating any capability as shipped.

## Project

`ootils-core` — a graph-based supply chain decision engine. FastAPI REST API on top of a Python kernel that models supply chains as typed nodes + edges, persisted in PostgreSQL 16. Core capabilities: incremental propagation, shortage detection, MRP explosion, scenario branching (deep-copy fork — historically labelled "copy-on-write"; see [REVIEW-2026-05 R10](docs/REVIEW-2026-05.md)), RCCP, a ghost/virtual-supply engine, and a data quality (DQ) pipeline.

## North Star — Ootils is an agent-piloted supply-chain operating substrate

**This is the single most important framing.** Ootils is not an APS with an AI bolt-on. It is the **deterministic substrate** on top of which a fleet of agents (watchers, scenario workers, governance, orchestrators) continuously monitors, diagnoses, simulates, ranks, and drafts actions. Humans supervise exceptions and irreversible decisions; agents absorb the rest. Reference: `docs/STRATEGY-autonomous-supply-chain-operations.md`.

Every design and implementation decision must be evaluated against this lens:

- **Forkable / scenario-first** — every state-changing capability must work inside a scenario fork so agents can test counter-factuals without touching baseline. No feature is "agent-ready" if it only works on baseline.
- **Deterministic core, stochastic edge** — LLMs/agents never own core calculations. Engine is deterministic; agents propose, govern, approve. Reproducibility is non-negotiable.
- **Queryable from a scenario** — every read path (`GetNode`, `QueryShortages`, etc.) must accept a `scenario_id`. Agents read from forks, not just baseline.
- **Streamable** — agents subscribe to deltas, they don't poll. Concrete rule (ADR-027, #391): every state-changing capability emits a typed event into the `events` table (migrations 002/006/051), which feeds `GET /v1/stream` (SSE). Agents subscribe via `/v1/stream?cursor=<stream_seq>`; a change that does not write an `events` row is invisible to the stream. For governed recommendations the emission is automatic: `governed_run` calls `emit_recommendation_created_for_run` (AN-1, #401), which COUNTs new rows across `_RECO_TABLES` (`engine/events/emit.py`) by `agent_run_id` and emits ONE `recommendation_created` event iff ≥1 new row. **Any new governed-reco table MUST be added to `_RECO_TABLES`, else its watcher's recos are invisible to the stream** (current set: `recommendations`, `parameter_recommendations`, `forecast_drift_recommendations`).
- **Explainable** — every calculation must be traceable. Recommendations without evidence are rejected by governance agents.
- **Auditable** — every write (by agent or human) is logged with input, output, scenario_id, calc_run_id, policy result. Audit is a feature, not telemetry.
- **Confidence-aware** — outputs that agents consume (forecasts, anomalies, recommendations) must carry a confidence score and a data-freshness flag. Stale data or low DQ blocks autonomous actions.
- **Decision Ladder L0-L4** (cf. strategy doc §5) — every action is classified by reversibility/risk. L0-L2 may be autonomous; L3-L4 require human approval.
- **Budgeted / kill-switchable** — agent-facing endpoints must support idempotency, per-agent scopes, rate limits, and global kill switches.

**Anti-patterns to refuse** (even if requested):
- A module that only works on baseline (not forkable).
- A read endpoint without `scenario_id` parameter.
- A write that bypasses the recommendation/approval state machine for L3+ actions.
- A forecast / score / metric without a confidence or freshness signal.
- An LLM call inside a deterministic calculation path.
- A state-changing feature that writes no typed `events` row (invisible to `/v1/stream`), without audit log, without explanation trace.

**Wedge V1** : "Autonomous shortage control tower with scenario-backed recommendations." Every near-term feature is judged by whether it advances this wedge.

## Architecture

### Request lifecycle
HTTP request → Bearer-token auth (`api/auth.py`) → router in `api/routers/<domain>.py` → engine call → `db/connection.py` yields a `psycopg` connection (autocommit-on-success, rollback-on-exception, `dict_row` factory). One router per capability domain; routers are thin and delegate to the engine.

### Engine layout (`src/ootils_core/engine/`)
- `kernel/graph/` — `store` (CRUD over nodes/edges), `traversal` (topological + subgraph expansion), `dirty` (dirty-flag manager).
- `kernel/calc/` — `projection` (ProjectionKernel), `calendar`.
- `kernel/shortage/`, `kernel/explanation/`, `kernel/allocation/`, `kernel/temporal/` — specialized kernels.
- `orchestration/propagator.py` — `PropagationEngine.process_event()` is the main entry point: acquires advisory lock → expands dirty subgraph → topo sort → compute → persist → cascade. Pairs with `orchestration/calc_run.py` which tracks run status.
- **Propagation has FOUR flavours, selected by `OOTILS_ENGINE` — know which one a given code path actually reaches before touching propagation behaviour** (dispatch in `api/routers/events.py:_build_propagation_engine`, `events.py:24-94`):
  1. `sql` — **default since 2026-05-24** (`events.py:56`; never overridden by `docker-compose.yml`/`Dockerfile`) — `propagator_sql.py`'s `SqlPropagationEngine`, window-function projection inside Postgres. See `docs/PERF-BASELINE.md`.
  2. `python` — the original in-process pure-Python kernel (base `PropagationEngine`), kept for parity testing (`scripts/parity_sql_vs_python.py`) and last-resort fallback.
  3. `rust` — opt-in, **Architecture A** (ADR-016, PyO3 in-process). `propagator_rust.py`'s `RustPropagationEngine` overrides **only** `_propagate()` — below `OOTILS_RUST_MIN_DIRTY` dirty PIs it falls back to inline SQL (default 0 = always Rust, `propagator_rust.py:58-68`). Same contract as the SQL engine otherwise: shortage detection still runs `SHORTAGES_SQL` on Python's session afterwards (`propagator_rust.py:218-232`, so ADR-021's canonical `shortages` persistence and ADR-025's overlay-aware resolution both apply unchanged), same audit/`events`/calc_run lifecycle. Needs the compiled `ootils_kernel` wheel (`WITH_RUST=1` in Docker, or local `maturin build --release`).
  4. `rust-svc` — opt-in, **Architecture B** (ADR-017, standalone gRPC service, in-RAM graph). `propagator_rust_svc.py`'s `RustServicePropagationEngine` overrides `process_event()` **wholesale**, not `_propagate()`. THREE known blockers keep it non-default: (a) it **never writes the `shortages` table** — detection stays purely in-RAM, `SHORTAGES_SQL`/`CLEAR_DIRTY_SQL` are explicitly skipped (`propagator_rust_svc.py:51-55,156-159`), invisible to ADR-021's canonical persistence axis / `/v1/issues`; (b) `_propagate()` is **not overridden**, so the two call sites that invoke it directly instead of `process_event()` — `calc.py:93` (`full_recompute`) and `simulate.py:234` — silently fall through to the base **pure-Python** engine under `rust-svc`, a correctness/perf trap with no error surfaced; (c) the in-RAM baseline graph loads once at boot with **no reload/refresh RPC** in the contract (`rust/ootils_proto/proto/engine.proto` defines no such method), so a Postgres write from any other engine or process while the service runs is invisible until restart.
  **Invariant: the default does not flip without an ADR.** `OOTILS_ENGINE` stays `sql`; the roadmap trigger to reconsider is **SCALE-2** (`docs/ROADMAP-AGENTS-2026-H2.md` §6, arbitration not yet decided). Docker: the `ootils_kernel` wheel (flavour 3) is compiled only when `WITH_RUST=1` (`Dockerfile`, BuildKit stage-selection — `WITH_RUST=0` never pulls `rust:1.82-slim`); the standalone service (flavour 4) ships via the separate `Dockerfile.engine`, unrelated to `WITH_RUST`.
- `scenario/manager.py` — scenario forking via deep-copy (the original CoW vocabulary doesn't match the implementation; see REVIEW-2026-05 R10).
- `dq/`, `ghost/` — capability modules under `engine/`; the `dq/agent/` subtree is an LLM-driven remediation agent.
- **Forecast confidence is REAL, not decorative (ADR-023):** `pyramide/confidence.py` is the single pure/deterministic composer (backtest WAPE x history depth x `demand_history.ingested_at` freshness; missing component → prudent 0.25 default, traced `components` + `stale` — never an optimistic 1.0). (3) **Reco → outcome chaining** (`engine/outcome/evaluator.py`, migration 069) — `evaluate_outcome` is a PURE 5-way classifier (AVOIDED / MATERIALIZED / PARTIAL / NOT_APPLICABLE / INDETERMINATE), thresholds 🎯-tunable in the evaluator (`AVOIDED_EPS_RATIO`/`AVOIDED_EPS_ABS`/`MATERIALIZED_FLOOR_RATIO`), NOT_APPLICABLE = counter-factual of never-approved recos (cost-of-inaction, no credit), INDETERMINATE = no snapshot, `avoided_severity_usd` NULL-honest; read-only on recommendations/shortages/inventory_snapshots, writes ONLY `recommendation_outcomes`; **baseline-only because an outcome is the REAL observed result — a fork is simulated, not observed**; `scenario_id` INHERITED via `recommendation_id` (no redundant column → no `scenarios` FK → stays out of the `test_scenario_fk_retention` guard). The **5 proof KPIs** (`GET /v1/outcomes/summary`, `api/routers/outcomes.py`): pct_shortages_avoided, avoided_severity_usd_total, avg_fva_wape (the real FVA), reco_approval_rate, cost_of_inaction_usd — each NULL/0-honest (`*_basis_count` distinguishes "no data" from a genuine zero). Classifier thresholds AND the flagship-KPI choice are 🎯-tunable pilot knobs. ### Storage
- PostgreSQL 16 via `psycopg[binary]` 3.x — **not** SQLite (ADR-005 proposes SQLite but the project has moved past the proof stage).
- UUID PKs, `TIMESTAMPTZ` UTC, **no JSONB** for business data. The "JSONB carve-out" pattern: diagnostic / forensic payloads with unbounded shape are the only acceptable JSONB sites, and each must carry a top-of-file comment block explaining the rationale (see `db/migrations/012_dq_agent.sql`, `021_mrp_lot_sizing_params.sql`, `031_demo_runs.sql`). Today's carve-out list: `dq_agent_runs.summary`, `mrp_runs.errors`, `mrp_runs.warnings`, `demo_runs.artifact`. Every other column uses typed columns. 73 numbered SQL migrations under `src/ootils_core/db/migrations/`.
- Migrations auto-apply on `OotilsDB()` construction (i.e. at API startup), serialized by a PG advisory lock (`_LOCK_KEY = 8_037_421_901`), tracked in `schema_migrations`. Each migration runs in **its own transaction** (`conn.transaction()`, `db/connection.py:181-197`); on **any** error the runner logs it, the transaction rolls back, and the exception **propagates** — the migration is **NOT** recorded as applied, and the process **crashes at boot**. The runner does *not* swallow "already exists"-family errors. Consequence: defensive idempotence (`IF NOT EXISTS` / `CREATE OR REPLACE` / `DROP … IF EXISTS`) is a **necessity**, not a nicety. The whole file is one transaction, so a statement failing mid-file rolls back the earlier ones too; but a fixed migration re-attempted at the next boot still re-runs every statement from scratch, and each must be a clean no-op on objects a prior partial run may have created. See migration 063's header block for the canonical defensive-idempotence pattern.
- `events` are conceptually insert-only for the **payload**, but mutable for **bookkeeping metadata** (`processed` flag, `updated_at`). Sites that update `processed = TRUE` in the orchestration layer (`engine/orchestration/propagator.py`, `engine/orchestration/calc_run.py`) are by design — they advance the event lifecycle without rewriting the event. ADR-005 D2's "insert-only" applies to the payload, not the flag.
- **Site aliases (ADR-031, migration 070):** a location carries N source-system codes via `location_aliases`; the resolution semantics (`external_id ∪ aliases`) live in ONE helper — `pyramide/repository.py:_warehouse_codes_subquery()` — never hand-write a `demand_history.warehouse_id = locations.external_id` equality (it misses aliases and reopens the #408 gap). The hard invariant — one code resolves to exactly ONE site, across ALL source_systems — is enforced at ingest (`POST /v1/ingest/locations`, symmetric 422s), NOT by a DB constraint: never INSERT into `location_aliases` directly.
- **Fork purge + shortage retention (ADR-039, PURGE-1, migration 076):** closes the follow-up ADR-011 explicitly left open — an archived scenario (`status='archived'`, every FK to `scenarios` `ON DELETE RESTRICT`) never had a reclaim path for its deep-copied child rows. `engine/maintenance/purge.py` adds two INDEPENDENT TTL-driven sweeps, each split `plan_*` (SELECT-only) / `apply_*` (sole writer, never commits — caller owns the transaction, re-verifies every absolute guard on FRESH data, never trusts a possibly-stale plan): (1) **fork purge** — deletes an archived, non-baseline, TTL-elapsed (`--ttl-days`, default 7, 🎯) scenario's child rows through the FK-ordered `PURGE_WHITELIST` tuple, **never** the `scenarios` row itself (`purged_at` stamped — the ADR-011 tombstone, no hard-delete); (2) **shortage retention** — deletes long-`resolved` `shortages` rows past `--retention-days` (default 30, 🎯), across every scenario (archived or not), hardcoded to skip `status='active'` and each scenario's own latest `completed` calc_run — a bounded GC delegated from `ShortageDetector`, which stays the exclusive writer of shortage semantics (ADR-021 amended). `tests/test_purge_whitelist_guard.py` re-derives every scenario-scoped table straight from the migration files on disk and fails the build if a new one is neither in `PURGE_WHITELIST` nor justified (≥20 chars) in `PURGE_EXEMPT_TABLES` — the governed-recommendation audit family (`recommendations`, `agent_runs`, `parameter_recommendations`, `dq_findings`, `eando_recommendations`, `forecast_drift_recommendations`, `scenario_promotions`) is deliberately exempted, never purged. Emits exactly ONE `purge_executed` event per run (ADR-027 run-granularity), stamped AFTER the `events` table's own delete so the confirmation event survives its own scenario's purge — the one sanctioned ADR-005 carve-out (events are payload-insert-only; a purged scenario's own events are the exception, never baseline/live-scenario events). Kill switch `OOTILS_PURGE_ENABLED`, default OFF (destructive-adjacent, opt-in unlike the read-mostly switches elsewhere). V1 ships the CLI (`scripts/purge_maintenance.py`, dry-run by default, `--apply` gated a SECOND time by the kill switch) and the read-only `GET /v1/maintenance/purge-preview` (`admin` scope) — **no HTTP apply endpoint** (architect decision: a destructive FK-ordered multi-table delete is an operator-run CLI job, never a POST); recurring production execution (cron/systemd timer) is the intended cadence but is not wired in this PR.

### Auth
`api/auth.py` validates `OOTILS_API_TOKEN` at **import time** and raises `RuntimeError` if unset. Token comparison uses `hmac.compare_digest`. Don't add an "optional auth" path.

**Scopes & budgets (ADR-032, #392 AN-2):** a whitelist of 8 scopes lives in `VALID_SCOPES` (`api/auth.py`) — `read`, `ingest`, `calc:run`, `graph:write`, `scenario:write`, `recommend:draft`, `recommend:approve`, `admin` (superset). Every mounted route now depends on `require_scope("...")`; there is **no `require_auth` on any write** (it survives only as a thin alias for tests / the unmounted `atp/api.py`). Doctrine is **cost ≠ reversibility**: any endpoint that invokes an engine (MRP, propagation, Pyramide, DQ, CTP, ghosts, BOM explosion) requires `calc:run` including its derived graph writes; `graph:write` is only for direct master-data mutations (firm/unfirm). Per-token `rate_per_min` is enforced (`_RateCounter`/`_enforce_rate_limit`, per-worker sliding 60 s window, 429 + `Retry-After`; legacy & NULL exempt). Token lifecycle is `POST/GET/DELETE /v1/tokens` (admin; cleartext shown once; soft-revoke + global cache invalidation); `GET /metrics` (admin, outside `/v1` and OpenAPI, bounded cardinality).

### Human window (ADR-036, EXP-1 PR1)
`GET /ui` (`api/routers/ui.py`) is the one arbitrated exception to CONTRIBUTING.md's "API first, UI never" — a server-rendered, read-only shell (`console.html`, Jinja2Templates, autoescape on) that is a **client of the API, never a privileged path**: no auth on the route itself (like `/health` — a plain navigation carries no `Authorization` header), zero business data in the render context, and no second auth mechanism (no cookie, no server-side token storage). The operator's token lives only in the browser's `sessionStorage` and rides `Authorization: Bearer` on every `fetch()` from the vanilla-JS static client (`api/ui/static/app.js`) against **existing** `/v1/*` endpoints — PR1 wires 3 read-only panels (recommendations inbox grouped client-side by `decision_level`, the 5 `GET /v1/outcomes/summary` proof KPIs rendered `None`-honestly as `"n/a"`, and `GET /v1/scenarios/compare`); no action buttons yet (PR2). Kill switch `OOTILS_UI_ENABLED`, **default OFF** (🎯 pilot call — the demo flips it), evaluated **once at `create_app()`**, not per-request — disabled means the route and its `/ui/static` mount are never registered (clean 404), unlike the per-request 503 pattern elsewhere (`outcomes.py`, `scenarios.py`'s `/compare`). `GET /v1/whoami` (`api/routers/me.py`, ordinary `/v1/*` route, `require_scope("read")`) serializes the resolved `Principal` (`actor_kind`, `scopes`, non-secret `token_prefix`) — **never the token** — consumed cosmetically by the page to decide which affordances to render; the server stays the sole enforcer via `require_scope` on every real write. CSP carve-out in `api/app.py`'s security-headers middleware: `/ui` and `/ui/*` get `default-src 'self'; frame-ancestors 'none'` with **no `'unsafe-inline'`** (console.html has zero inline `<script>`/`<style>`/`onclick=`), distinct from the relaxed `/docs`/`/redoc` CSP branch. See ADR-036 for the full decision record, including the rejected cookie-session alternative and PR2/graph_fragment follow-ups.

### Feed contracts & governed ingest (ADR-037, INT-1 PR1)
`feed_contracts` (migration 073) is a versioned, append-only-per-version registry of what a source feed (on-hand, open POs, open WOs, …) is allowed to look like — entity mapping, format, business key, mandatory columns, cadence, arrival window, owner, `criticality` (`blocking`/`advisory`), volume guards. `config/feed-contracts/*.yaml` is the pilot-editable source of truth (3 seed contracts, heavily commented for the pilot to fill in); `src/ootils_core/interfaces/contracts.py` (`FeedContractSpec`, strict Pydantic, `extra="forbid"`) parses it and `scripts/load_feed_contracts.py` loads it idempotently — identical content is a traced no-op, any diff mints a new version, an existing version's content is **never** `UPDATE`d. `entity_type`/`format` CHECKs are deliberately kept in lockstep with `ingest_batches_entity_type_check` (023→035→036) and `staging.uploads.file_format` (033) — widen those first, then mirror here. `load_mode` accepts only `'full'` in V1 (fail-loudly at both the Pydantic and the DB CHECK layer; delta is V2 scope). **PR1 is registry-only — nothing reads this table at ingest time yet.** This is also the chantier that supersedes ADR-013 D4 (mandatory human approval) for the *governed daily-run* case: the pilot-decided doctrine (2026-07-11) is auto-approve iff DQ green AND all guards green, with a red guard on a `blocking` feed escalating via the existing `notifications/l3_webhook.py` L3 webhook instead of blocking silently — wired in PR2/PR3, not in PR1. See ADR-037.

**Daily-run pipeline is real (ADR-042, PR-4a/4b/4c):** `engine/ingest/daily_orchestrator.py` (scan → resolve contract → observe → decide → gate-then-load, reusing `interfaces/ingest_exec.py`'s parse/build/call/archive primitives) and its CLI `scripts/run_daily_ingest.py` (dry-run by default; `--apply` gated by `OOTILS_DAILY_RUN_ENABLED`, default OFF) actually scan an inbox and load feeds; `engine/reporting/daily_report.py` renders a deterministic Markdown compte-rendu (`daily_report_<date>.md`) deposited into `dropbox:ootils-outbox` by the separate, dumb `scripts/deposit_outbox.sh` (one-way `rclone copy`), and `GET /v1/daily-runs` (`api/routers/daily_runs.py`, kill switch `OOTILS_DAILY_RUN_REPORT_ENABLED`, default ON) reads back the guard history + governed decision. Because `dq_status_by_feed` still isn't DB-wired (that's ADR-042's PR-1, "Fencer l'ingest direct", not yet delivered), every real V1 run honestly ceilings at `DEGRADED` — never a fabricated `AUTO_APPROVED` — see ADR-042's 2026-07-18 amendment.

### Propagation model (ADR-003)
Event-driven, incremental, deterministic. An event marks a subgraph dirty; compute happens in topo order; unchanged nodes stop the cascade (they do not propagate further). Every change is attributable. Determinism is a hard constraint — **no randomness in the core engine**.

**Planner-stats invariant (#455):** any bulk INSERT into a table the planner reads back *within the same request* must be followed by an explicit `ANALYZE`. Canonical site: `DirtyFlagManager.flush_to_postgres` (`engine/kernel/graph/dirty.py`) runs `ANALYZE dirty_nodes` after the batch INSERT — without it, `PROPAGATE_SQL`/`SHORTAGES_SQL` see stale `rows=1` stats, the planner picks a per-row nested loop, and full propagation collapses to O(N²) (measured 272× slower: 1 099 s vs 4 s on 47 520 PI). All propagation flavors (sql/python/rust in-process) go through this site; rust-svc computes in RAM and is out of scope. Perf note (`docs/PERF-BASELINE.md` § re-bench 2026-07-11): once stats are fresh, **SQL and in-process Rust are within ±4%** end-to-end — the "Rust wins at every scale" framing was an artifact of the stale-stats bug; Rust's real leverage is the in-RAM rust-svc architecture (SCALE-2), not per-node compute.

## Conventions that aren't obvious from the code

- **Tests run against real Postgres, no mocks.** Point `DATABASE_URL` at a throwaway DB. Pure-Python helper functions (and Pydantic-validation 422 boundary tests) live in `tests/test_*.py` and don't need a DB. DB-touching tests live in `tests/integration/test_*_integration.py` and use the `conn` / `seeded_db` fixtures.
- `tests/legacy/` is intentionally excluded via `tests/conftest.py:collect_ignore_glob` — targets the pre-graph architecture, do not re-enable.
- `print()` is forbidden in production code paths — use the module `logger`. The exception is documentation: example `print()` calls **inside docstrings** (showing how a caller would use the returned value) are fine. See `src/ootils_core/forecasting/engine.py:80-81` for the canonical case.
- `/v1/ingest/*` has a 10 MB request-body cap enforced by `IngestPayloadSizeLimitMiddleware` in `api/app.py`.
- The generic exception handler in `api/app.py` deliberately hides exception strings from clients (logs them instead) to avoid leaking DSNs / stack traces. Don't "improve" it by echoing `str(exc)` to the response. **Typed domain exception carve-out**: a few routers do `raise HTTPException(detail=str(e))` where `e` is a *named domain exception* (`DiffError`, `ApprovalError`, `RejectionError` in `api/routers/staging.py:243,325,390`). These exceptions are raised by our own code at `staging/diff.py`, `staging/approve.py`, `staging/reject.py` with hand-authored messages that contain only UUIDs and status enums (no DSN, no DB error, no path) — the messages are part of the API contract and clients need them to act (e.g. "batch in terminal status `'imported'`; /diff only meaningful in 'pending'/'validated'"). The carve-out is limited to those three sites; any new `detail=str(e)` outside the named-domain-exception pattern is a real leak risk.
- Principles from `CONTRIBUTING.md` that the code enforces: API-first (no UI features in V1), explainability (every calculation traceable — see `kernel/explanation/`), fail-loudly over silent wrong answers.
- **`pyramide_runs.status` is `generated` | `committed` | `failed` — NOT `completed`** (migration 038 CHECK). A run is `generated` at persist time and only later `committed` (`repository.persist_run` → `_commit_snapshot_to_demand_nodes`); readers exclude only `failed` (so both `generated` and `committed` count). Separately, the AGGREGATE (pooled) backtest row in `pyramide_accuracy_metrics` is the one with **`horizon IS NULL`** — per-horizon rows carry an integer horizon. The demand watcher and the latest-aggregate readers join on `m.horizon IS NULL`; using `status='completed'` or forgetting the `horizon IS NULL` filter silently returns nothing.
