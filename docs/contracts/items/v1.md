## 2. Cadence et SLA

| Profil | Fréquence | SLA freshness | Justification |
|---|---|---|---|
| `DAILY` par défaut | 1 push / 24 h | 24 h | Master data items change peu ; un push nocturne par ERP suffit |
| `ON_CHANGE` (push immédiat) | événementiel | 1 h | Pour NPI, EOL, status flips |
| `BACKFILL` (initial load) | one-shot | – | Première synchro, mode `BACKFILL` explicite **[GAP]** |

→ Stocké dans `import_contracts (source_system, entity, version, sla_freshness_s)` **[GAP — table à créer]**.

Watchers concernés :
- **I-W01 Interface Health** : surveille latence, retry rate
- **G07 Import Freshness Gate** : bloque recommandations si feed items stale > SLA

## 3. Granularité

**DELTA** : upsert par `external_id` (= `source_ref`).

- Pas de SNAPSHOT (un push partiel ne désactive pas les items absents — sinon trop de risque opérationnel)
- Pas d'EVENT (un item est un état, pas un événement)
- Pas de CDC (V2 pour ERP qui poussent du CDC natif)

## 7. Validation

### 7.1 Structurelle (V1.0 actuelle)

- `external_id`, `name` non-vides (`field_validator non_empty`)
- `item_type` ∈ `{finished_good, component, raw_material, semi_finished}`
- `status` ∈ `{active, obsolete, phase_out}`
- All-or-nothing : toute erreur ligne → HTTP 422, **rien n'est persisté**

### 7.2 Business rules (V1.1 proposées) **[GAP]**

| Règle | Niveau | Owner |
|---|---|---|
| `uom` doit exister dans `uom_conversions` ou être un code SI standard | bloquant | DQ Watcher |
| Si `lifecycle_stage = EOL` alors `status = phase_out` ou `obsolete` | warning | End-of-Life Agent |
| Si `lifecycle_stage = NPI` alors `status = active` | warning | NPI Agent |
| Si `abc_class = A` alors avoir un `family` non-null | warning | DQ Watcher |
| `hs_code` valide (regex `[0-9]{6,10}`) si présent | bloquant | DQ Watcher |
| `country_of_origin` valide ISO 3166-1 alpha-2 si présent | bloquant | DQ Watcher |
| `shelf_life_days > 0` si `lot_tracked = true` | warning | DQ Watcher |
| Pack `case_pack >= pack_size` si les deux présents | warning | DQ Watcher |

### 7.3 Référentielles (FK) **[GAP partiel]**

Aucune FK directe en V1.0 (items est root). Les FK inverses arrivent :
- `bom_lines.parent_item_id` / `child_item_id` → items
- `supplier_items.item_id` → items
- `item_planning_params.item_id` → items
- `nodes` (via `item_id`)

Si un item passe `status = obsolete` mais a des nodes actifs → **warning DQ + propagation Lifecycle Watcher**.

## 8. DQ checks

### 8.1 À l'ingress (bloquants)

Cf. §7.1 (validation structurelle).

### 8.2 Post-load (non-bloquants, alimentent watchers)

Exécutés par `_trigger_dq(db, batch_id)` après commit (cf. `ingest.py:542`).

| Check | Agent consommateur | Sévérité |
|---|---|---|
| Items sans BOM mais marqués `finished_good` ou `semi_finished` | BOM Health (W11) | medium |
| Items `lot_tracked` sans `shelf_life_days` | DQ (W07) | medium |
| Items créés mais aucun `supplier_item` associé | Supply Watcher (W03) | low |
| Items dont `lifecycle_stage` n'a pas bougé depuis > 24 mois | NPI/EOL Agents | low |
| Items dont `abc_class` est null après 90 j d'historique | ABC/XYZ Drift (P-W04) | medium |
| Mass change détecté (>10% du catalogue en 1 batch) | Parameter Change Audit (P-W10) | high |

## 9. Mapping (codes externes → internes)

### 9.1 V1.0 (existant)

`external_id` = clé naturelle directe. Pas de table de mapping séparée.

### 9.2 V1.1 proposé **[GAP]**

Pour supporter **plusieurs sources** (ERP_SAP + WMS_X qui ont leurs propres codes pour le même item) :

```sql
CREATE TABLE external_code_mapping (
  source_system   TEXT NOT NULL,
  entity          TEXT NOT NULL,   -- 'items'
  external_code   TEXT NOT NULL,
  internal_id     UUID NOT NULL,   -- item_id
  status          TEXT NOT NULL DEFAULT 'active',  -- 'active' | 'deprecated' | 'merged'
  created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
  PRIMARY KEY (source_system, entity, external_code)
);
```

Politique fallback (code source inconnu) : `QUARANTINE` (cf. WIP-inbound §I3).
Le **Mapping Repair Agent (I-S02)** proposera une résolution.

## 10. Emit (events publiés vers agents)

### 10.1 V1.0 (existant)

- DQ engine triggered post-batch
- Aucun event applicatif structuré **[GAP]**

### 10.2 V1.1 proposé **[GAP]**

| Topic | Quand | Payload minimal |
|---|---|---|
| `import.batch.completed` | toujours | `{batch_id, entity, source_system, inserted, updated, errors, dq_status, correlation_id}` |
| `import.batch.failed` | si rejet 422 | `{batch_id, entity, source_system, errors[]}` |
| `items.upserted` | par lot (résumé) | `{batch_id, inserted_count, updated_count, item_ids[]}` |
| `items.lifecycle_changed` | si `lifecycle_stage` ou `status` modifié | `{item_id, old_value, new_value, source_system}` |
| `items.classification_changed` | si `abc_class` / `xyz_class` modifié | `{item_id, axis, old, new}` |

**Pas de `propagation.triggered`** — items est master data, pas un node graphe. Mais un changement de `status = obsolete` peut nécessiter un trigger downstream (nodes → DQ flag) à arbitrer.

## 14. Réponse standard

```json
{
  "status": "ok|dry_run|partial",
  "summary": {
    "total": 1234,
    "inserted": 1200,
    "updated": 30,
    "errors": 4
  },
  "results": [
    {"external_id": "SKU-001", "item_id": "uuid", "action": "inserted"},
    {"external_id": "SKU-002", "item_id": "uuid", "action": "updated"}
  ],
  "batch_id": "uuid",
  "dq_status": "passed|warning|failed"
}
```

Code retour :
- `200 OK` — accepté (full ou partial)
- `400 Bad Request` — Idempotency-Key malformé
- `401 Unauthorized` — token invalide
- `409 Conflict` — Idempotency-Key déjà utilisée avec payload différent
- `413 Payload Too Large` — body > 10 MB
- `422 Unprocessable Entity` — validation échouée (rien persisté)
- `429 Too Many Requests` — rate limit (V1.1 **[GAP]**)
- `500 Internal Server Error` — bug serveur
